# Commitment Decomposition

Scope: cloud models only. This continuation assumes Branch Commitment is real and tests whether it is primitive or composed of deeper runtime components. No external sciences, primitive searches, or new theory families are introduced.

## Decomposition Result

Branch Commitment decomposes into a sequence of observable runtime components:

`evidence accumulation -> confidence accumulation -> option elimination -> uncertainty collapse -> action lock-in -> verification completion / terminalization`

The sequence is not always complete. Successful runs usually show evidence converted into grounded action before lock-in. Failed runs often show action lock-in without adequate evidence-action conversion. Premature commitments compress the sequence too early. Delayed commitments extend evidence accumulation and option elimination without decisive action lock-in.

## Component Roles

| component | role in commitment | evidence status | determination |
| --- | --- | --- | --- |
| evidence accumulation | makes branch selection possible | strongly upstream through retrieval/evidence/grounding rows | necessary input, not the trigger alone |
| confidence accumulation | stabilizes the chosen branch | inferred from reduced branch switching and higher predictability | supporting component |
| uncertainty collapse | observable signature that outcome becomes easier to predict | directly measured at 50% prefix | diagnostic marker, not sole cause |
| search exhaustion | weak proxy for no remaining attractive alternatives | inferred from persistent non-switching trajectories | partial component |
| option elimination | removes competing branches | inferred from transition into grounded/converging states | close to trigger but not sufficient alone |
| action lock-in | makes the branch hard to redirect | directly measured by branch lock-in and reversibility | immediate commitment trigger |
| verification completion | terminalizes the branch or fails to reopen it | weak direct measurement in current trajectory data | late stabilizer |

## Trigger Determination

The component that most directly triggers commitment is **action lock-in after option elimination**.

Evidence accumulation and confidence accumulation prepare commitment, but they do not themselves commit the run. Uncertainty collapse is the measurable external sign that commitment has occurred. Search exhaustion and option elimination narrow the branch set. The transition into commitment happens when the remaining branch is converted into an action path that is no longer being actively revised.

Operationally:

`Branch Commitment onset = first point where a grounded or selected branch becomes action-dominant and branch switching/reversal begins to fall.`

## Successful vs Failed Commitment

| commitment type | component pattern | outcome reading |
| --- | --- | --- |
| successful commitment | evidence accumulates, action remains linked to evidence, options narrow, lock-in lands on grounded/converging path | commitment is useful |
| failed commitment | evidence is absent, misread, or disconnected; lock-in lands on stuck/wrong path | commitment is false or damaging |
| premature commitment | action lock-in occurs before evidence-action conversion is adequate | commitment is too early |
| delayed commitment | evidence and option narrowing occur, but lock-in remains reversible or indecisive too long | commitment is too late or underpowered |

## Main Finding

Branch Commitment is not a single primitive event. It is a runtime phase transition produced by narrower components, especially evidence-action conversion, option elimination, and action lock-in. However, it remains a dominant mechanism because those components become outcome-decisive only when they combine into a branch that is hard to reverse.
